How this needs to actually work

This page is a static mockup. Here's what production requires — read before building.

Pricing

Prices live in a database table (destination, government fee, service fee, updated_at). This admin page reads and writes to it through an authenticated API. Any change here should also be logged — who changed a fee and when — since it directly affects money customers are charged.

How AI document review actually runs

  • Customer uploads a file → it goes to private cloud storage (not public), e.g. an S3-compatible bucket.
  • A backend job sends the file to a vision-capable AI model (e.g. Claude) with a prompt describing the embassy's rules for that document type.
  • The model's structured response (pass / needs-review / fail + reason) is stored against the application record.
  • "Needs review" items surface in this queue for a human specialist — the AI never auto-rejects or auto-files an application on its own.
  • The API key for the AI model lives only on the backend server, never in the browser.

Where data is stored

  • Documents — encrypted object storage, access-controlled per application, retained per your data-retention policy (and UK GDPR — customers can request deletion).
  • Application & customer data — a proper database (e.g. Postgres), not spreadsheets — this is what powers the customer dashboard and this admin queue.
  • Payments — never store card details yourself; use a processor (Stripe etc.) and store only their reference IDs.

Access control

This admin page needs real authentication (staff login, not a public URL) and role-based permissions before it touches real prices or real customer documents. None of that exists in this prototype yet.
